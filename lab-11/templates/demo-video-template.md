## Why the Demo Video Matters

Demo Day judges see 10+ pitches in a row. Your demo video:
- **Proves your product actually works** (not just mockups or ideas)
- **Shows the problem being solved** (not just features)
- **Keeps attention** (visual beats talking)
- **Provides backup** (if live demo fails)
- **Demonstrates execution** (you built something real)

**Bad demo = "This is cool, but does it work?"**  
**Good demo = "Wow, I need this!"**

**Key Principle:** Show, don't tell. 60 seconds of product in action beats 10 minutes of explanation.

## Demo Video Formula

### Structure (60-90 seconds)

```
[0-10s]   HOOK: Show the pain / Set up the problem
[10-50s]  FLOW: Solve it step-by-step with your product
[50-70s]  PROOF: Show the result / benefit
[70-90s]  CTA: Call to action (what to do next)
```

### Script Template

```markdown
## [0-10s] Hook - Make Them Care

"[Customer name] wastes [X minutes] doing [painful task]. 
Watch how [Product Name] solves this in [Y seconds]."

OR

"Every day, [X thousand] [target users] struggle with [problem].
Here's how [Product Name] fixes it."

## [10-50s] Core User Flow - Show Don't Tell

[Screen: Homepage/Login]
"Open [Product Name]."

[Screen: Key Action 1]
"[Describe what's happening]. Notice [key feature working]."

[Screen: Key Action 2]
"[Next step]. The system automatically [smart thing product does]."

[Screen: Key Action 3]
"[Final step]."

[Screen: Result]
"Done. [X seconds] total, compared to [Y minutes] the old way."

## [50-70s] Key Benefits - Why It Matters

"No more [old painful way]. 
[Product] saves [time/money/hassle], 
[benefit 2], and [benefit 3]."

## [70-90s] CTA - What Next

"Try [Product] at [URL]. 
[Product tagline - one sentence]."
```

## Recording Options

### Option A: Screen Recording (Recommended)

**Best for:** Web apps, mobile apps (using simulator/emulator), dashboards, SaaS products

#### Mac Users: QuickTime (Built-in, Free)

**Steps:**
1. Open QuickTime Player
   - Press Command + Space
   - Type "QuickTime"
   - Press Enter
2. Click File ‚Üí New Screen Recording
3. Click options dropdown:
   - Microphone: Choose if narrating live
   - Click quality: Maximum
4. Click red record button
5. Select area to record:
   - Click for full screen
   - Drag to select region
6. Walk through your app (follow your script)
7. Click stop button in menu bar (or Command + Control + Esc)
8. File ‚Üí Save
9. Name it: demo-video.mov

**Tips:**
- Test audio before full recording
- Close notifications (Do Not Disturb mode)
- Practice cursor movements

#### Windows Users: Xbox Game Bar (Built-in, Free)

**Steps:**
1. Press Windows + G (opens Game Bar)
2. Click "Yes, this is a game" if prompted
3. Click Capture widget (camera icon)
4. Click record button (or Win + Alt + R)
5. Walk through your app
6. Press Win + Alt + R to stop
7. Video auto-saves to: C:\Users\[YourName]\Videos\Captures
8. Rename to: demo-video.mp4

**Tips:**
- Check that Game Bar is enabled: Settings ‚Üí Gaming ‚Üí Xbox Game Bar
- Adjust quality: Settings ‚Üí Gaming ‚Üí Captures ‚Üí Video quality

#### Any Platform: Loom (Easiest, Free Tier Available)

**Steps:**
1. Go to loom.com
2. Sign up (free account: 25 videos, 5 min each)
3. Download desktop app
4. Open Loom app
5. Click "Start Recording"
6. Choose: Screen Only (or Screen + Cam if you want face)
7. Select window or full screen
8. Click "Start Recording"
9. 3-2-1 countdown
10. Walk through your demo
11. Click "Finish" in Loom toolbar
12. Video auto-uploads to cloud
13. Click "Download" ‚Üí Save as MP4

**Why Loom is easiest:**
- Auto-uploads (backup)
- Built-in trimming
- Easy sharing link
- No setup required

## What to Show

### ‚úÖ DO Show:

**1. The Pain First (3-5 seconds)**
- Start with the problem state
- Example: "Finding a study room usually takes 15 minutes. Watch this."
- Sets context for why your product matters

**2. One Complete User Journey**
- Beginning to end
- Actual problem being solved
- Real data (not "lorem ipsum" or fake content)
- Happy path (no errors)

**3. The Magic Moment**
- When your product does something smart/automated
- The "wow" feature that differentiates you
- What makes you better than alternatives

**4. The Result**
- Problem solved
- Time saved (quantify it!)
- User benefit achieved
- Before/after comparison

**5. Real Use Case**
- Scenario customers actually face
- From your interview data (Lab 2-4)
- Authentic, not contrived

### ‚ùå DON'T Show:

**1. Multiple disconnected features**
- ‚ùå "Here's the dashboard, here's settings, here's reports..."
- ‚úÖ Pick ONE complete flow that solves THE problem

**2. Bugs or incomplete features**
- Only show what actually works
- Polish the happy path
- Better to show less that works than more that's broken

**3. Lengthy setup or login**
- Start already logged in
- Skip account creation screens
- Jump straight to value

**4. Your face explaining (unless intentional)**
- Product should be the star
- Voiceover is fine
- But don't block the screen with your face

**5. Reading text from the screen**
- Viewers can read
- Narrate the ACTION, not the labels
- "Click 'Find Room'" not "Here's a button that says 'Find Room'"

## Recording Best Practices

### Before Recording

**1. Prepare Your Environment (5 minutes)**
- [ ] Close all unnecessary tabs/apps
- [ ] Turn off notifications
  - Mac: Control Center ‚Üí Do Not Disturb
  - Windows: Action Center ‚Üí Focus Assist
  - Phone: Do Not Disturb mode
- [ ] Clear browser history/autocomplete (no embarrassing suggestions)
- [ ] Hide sensitive info (emails, names, phone numbers)
- [ ] Use realistic test data (not "asdf" or "test user")

**2. Set Up Your Screen (3 minutes)**
- [ ] Window size: Not too small (text unreadable) or too large (wasted space)
- [ ] Zoom browser if needed: Command/Ctrl + (make text bigger)
- [ ] Resolution: 1080p minimum (1920x1080)
- [ ] Lighting: Good if showing phone or physical product
- [ ] Sound check: If doing voiceover, test microphone

**3. Practice Dry Run (10 minutes)**
- [ ] Click through flow 3-5 times
- [ ] Get smooth at cursor movements
- [ ] Memorize where to click (no hesitation)
- [ ] Time yourself (should be under 80 seconds)
- [ ] Note any steps that take too long

**Pro Tip:** Do dry run with teammate watching. They'll catch things you miss.

### During Recording

**Voice (if narrating):**
- Speak clearly and slightly slower than normal
  - You'll speed up when nervous
  - Give viewers time to process
- Pause briefly between sentences
  - Easier to edit later
  - Gives viewer breathing room
- Don't say "um," "uh," "so," "like"
  - Practice script beforehand
  - It's okay to pause silently
- Energy and enthusiasm!
  - You're excited about this product
  - Enthusiasm is contagious

**Mouse/Cursor Movements:**
- Smooth movements (not jerky or frantic)
- Move deliberately to what you're clicking
- Pause cursor on element for 1 second before clicking
- After clicking, pause to show result
- Don't circle or wiggle cursor randomly

**Pacing:**
- Not too slow (viewers get bored)
- Not too fast (confusing, hard to follow)
- Let each action complete before moving to next
- Show results of each action (confirmation messages, updates)
- Match your narration speed to screen speed

**Common Mistakes to Avoid:**
- ‚ùå Clicking before narrating what you're doing
  - ‚úÖ Say it THEN do it
- ‚ùå Moving too fast through screens
  - ‚úÖ Pause 2 seconds on each new screen
- ‚ùå Not showing the outcome
  - ‚úÖ After action, show what changed
- ‚ùå Apologizing or saying "oops"
  - ‚úÖ Just stop recording and start over

### After Recording

**1. Review Your Video (3 minutes)**
- [ ] Watch full video start to finish
- [ ] Check: Is text readable?
- [ ] Check: Is audio clear (if voiceover)?
- [ ] Check: Does it make sense without explanation?
- [ ] Check: Any awkward pauses or mistakes?
- [ ] Check: Under 90 seconds?

**2. Decide: Keep or Redo?**

**Keep it if:**
- Main flow is clear
- No major bugs visible
- Under 90 seconds
- Text readable
- Makes sense

**Redo if:**
- Major confusion in flow
- Bug or error visible
- Over 2 minutes (can't edit down)
- Text unreadable
- Audio has loud background noise

**Remember:** Perfect is the enemy of done. If it's 80% good, keep it and move on.

## Editing Your Video

### Simple Editing (Most People)

**Tool Options:**
1. **iMovie (Mac)** - Free, simple
2. **Windows Video Editor** - Free, built-in to Windows 10/11
3. **Kapwing (Web)** - kapwing.com, free tier
4. **Loom** - Built-in trimming

**Basic Edits You Might Need:**

**1. Trim Start/End**
- Cut first 3 seconds (before you start)
- Cut last 5 seconds (after CTA)
- Most tools: drag handles on timeline

**2. Cut Out Mistakes**
- Pause or loading screen too long
- Clicked wrong thing
- Most tools: split clip, delete section

**3. Speed Up Slow Parts**
- Loading screens
- Long forms being filled out
- Most tools: select section, speed 1.5x or 2x

**4. Add Text Overlays (Optional)**
- Key stats: "Saves 15 minutes"
- Feature labels: "Auto-generate reports"
- Most tools: text tool, place on screen

**5. Fade In/Out (Optional)**
- Smooth start/end
- Most tools: transitions section

### Do NOT Over-Edit

**You DON'T need:**
- Fancy transitions
- Background music (often distracting)
- Multiple camera angles
- Special effects
- Custom animations

**Simple is better.** Your product is the star.

### Export Settings

**Format:** MP4 (most compatible)  
**Resolution:** 1080p (1920x1080) preferred, 720p acceptable  
**Frame Rate:** 30fps is fine  
**Codec:** H.264  
**File Size:** Under 50MB (compress if larger)

Most tools have a "YouTube" preset that works great.

## Troubleshooting

### Problem: "My demo takes 3 minutes, not 90 seconds"

**Solution A: Cut steps**
- Show ONLY the core value flow
- Skip login/setup (start already in)
- Remove secondary features
- Focus on problem ‚Üí solution

**Solution B: Speed up video**
- Edit in video editor
- Speed up 1.25x or 1.5x
- Or speed up just the slow parts (forms, loading)

**Solution C: Two shorter videos**
- Make 45-second "Core Flow" video
- Make 45-second "Advanced Features" video
- Use core flow in pitch

### Problem: "My app has loading spinners / API calls"

**Solution A: Speed up during edit**
- Speed those sections 2x-4x
- Or cut them out entirely
- Jump from action to result

**Solution B: Pre-load data**
- Before recording, trigger API calls
- Then record with cached data
- Everything loads instantly

**Solution C: Use faster test environment**
- Local development (faster than production)
- Test database with less data
- Mock API responses

**Solution D: Add text overlay**
- During loading: text says "Loading..."
- Or: "Analyzing 10,000 records..."
- Makes it clear it's supposed to load

### Problem: "I'm not good at voiceover / I hate my voice"

**Solution A: Silent demo with text**
- No narration
- Use text overlays to explain each step
- Add text at top/bottom of screen

**Solution B: Multiple takes**
- Record narration 3-5 times
- Pick the best one
- Edit together best parts

**Solution C: Teammate does voiceover**
- Someone else on team narrates
- You click through the demo
- Record audio separately, sync later

**Solution D: Captions only**
- Record demo without sound
- Add captions in video editor
- YouTube-style format

### Problem: "My screen recording is laggy / low frame rate"

**Solution A: Close other apps**
- Quit everything except your MVP
- Restart computer first
- Don't record while running heavy apps

**Solution B: Record smaller window**
- Don't record full 4K screen
- Record just your app window
- Smaller area = better performance

**Solution C: Use different tool**
- Try Loom instead of built-in
- Or OBS Studio (free, powerful)
- Update graphics drivers

**Solution D: Record on better computer**
- Teammate's laptop
- School computer lab
- Friend's desktop

### Problem: "I don't have a working MVP yet"

**Solution A: Clickable prototype**
- Export from Figma as prototype
- Click through it
- Record as if it's real app
- Narrate what it WILL do

**Solution B: Screenshot video**
- Take 10-15 screenshots of key screens
- Import into PowerPoint/Keynote
- Add transitions (0.5s each)
- Animate between screens
- Export as video
- Add narration explaining flow

**Solution C: Partial demo**
- Show the parts that DO work
- For incomplete parts: "Coming soon: [feature]"
- Be honest: "Building now, launching [date]"

**Solution D: Competitor demo + explanation**
- Find similar product demo
- Show it briefly
- Explain: "Ours will do this, PLUS [your differentiation]"
- Only if you really have nothing

### Problem: "Demo has a bug / error that we can't fix in time"

**Solution A: Work around it**
- Demo different feature
- Use test account that doesn't trigger bug
- Start from middle of flow (skip buggy part)

**Solution B: Acknowledge it**
- Show demo
- If bug appears, text overlay: "Beta - fixing this week"
- Shows transparency
- Better than hiding it

**Solution C: Use older version**
- Revert to last working version
- Record demo with that
- Continue fixing current version

### Problem: "App looks ugly / unpolished"

**Solution A: Focus on functionality**
- Show that it WORKS
- Narration: "MVP focused on core functionality"
- Design polish is Week 13

**Solution B: Record just the polished screens**
- Skip screens with bad UI
- Show only the good parts
- Narration covers the gaps

**Solution C: One hour of polish**
- Fix just the colors/fonts on main screens
- Make buttons consistent
- Clean up alignment
- Don't redesign, just tidy
